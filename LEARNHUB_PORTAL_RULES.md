# LearnHub Portal Rules

Этот файл нужен как локальный ориентир для любых доработок `LearnHub Portal`.

## Область работы

- По умолчанию трогать только `LearnHub-Portal.html`.
- Не менять другие проекты в репозитории без явного запроса.
- Любая правка должна быть напрямую связана с задачей пользователя.

## Think Before Coding

- Не делать молчаливых допущений.
- Если есть несколько трактовок, сначала явно назвать их.
- Если решение можно сделать проще, выбирать простое.
- Если что-то непонятно, сначала зафиксировать неясность и только потом кодить.

## Simplicity First

- Писать минимальный код, который решает задачу.
- Не добавлять фичи, которые не просили.
- Не создавать абстракции “на будущее”, если они не нужны сейчас.
- Не переусложнять обработку невозможных сценариев.
- Если решение выглядит слишком тяжёлым, переписать проще.

## Surgical Changes

- Менять только то, что нужно для задачи.
- Не рефакторить соседний код без запроса.
- Сохранять существующий стиль файла.
- Удалять только тот мусор, который появился из-за текущих правок.
- Если найден старый мёртвый код, отмечать его отдельно, но не трогать без запроса.

## Goal-Driven Execution

- Перед реализацией формулировать проверяемую цель.
- Для многошаговых задач держать короткий план:
  1. что меняем
  2. как проверяем
  3. что считаем успешным результатом
- После правок всегда делать техническую проверку:
  - извлечь `text/babel`-скрипт из `LearnHub-Portal.html`
  - прогнать через `esbuild --loader:.js=jsx`

## Project-Specific Rules

- Не ломать совместимость с `localStorage`-ключами `lh_*`.
- Если меняется поведение курсов, шагов, прогресса или заявок, проверять:
  - запись на курс
  - прохождение шага
  - возврат назад
  - сохранение прогресса после reload
- Если меняется логика ролей или пользователей, проверять:
  - `users`
  - `roles`
  - `enrollments`
  - `submissions`
  - `stepProgress`
  - `stepAnswers`
- Если добавляется новый тип шага, он должен быть согласован в:
  - `STYPES`
  - `ICONS`
  - конструкторе
  - viewer
  - сохранении ответов

## UI Rules

- Для вопросов и тестов сохранять состояние ответа до reload.
- Не позволять кнопке `Далее` молча сбрасывать уже подтверждённый ответ.
- Для пользователя интерфейс должен быть понятен без чтения кода:
  - явные статусы
  - понятные кнопки
  - подтверждение необратимых действий

## Testing Checklist

- Регистрация нового пользователя.
- Запись на курс и статус `pending`.
- Подтверждение/отклонение заявки.
- Прохождение одиночного вопроса.
- Прохождение вопроса с множественным выбором.
- Пропуск вопроса.
- Открытый вопрос.
- Опрос.
- Возврат назад без потери подтверждённого ответа.
- Финальный экзамен.
- Сертификат и его шаринг.
