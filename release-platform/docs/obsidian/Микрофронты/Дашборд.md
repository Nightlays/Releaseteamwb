# Дашборд

## Назначение

Главный экран готовности релиза: собирает состояние Allure-прогонов, считает общий прогресс, ручную активность, uWu-остаток, readiness по платформам и прогноз риска/ETA.

## Где код

- UI: `src/modules/Dashboard/index.tsx`
- Реестр: `dashboard` в `src/config/modules.tsx`
- Supabase persistence: `src/services/dashboardSupabase.ts`
- Allure aggregation: `src/services/allure.ts`
- Prediction: `src/services/releasePrediction.ts`
- Historical baseline: `src/services/regressionHistory.ts`
- DB schema: `supabase/dashboard_snapshots.sql`
- Legacy fallback: `legacy/dashboard_run_module.html`

## Данные и сервисы

- [[Allure TestOps]] через `src/services/allure.ts`:
  - `fetchDashboardAggregate`
  - `fetchReadinessSummary`
  - dashboard cache helpers
- `src/services/releasePrediction.ts`:
  - `buildDashboardPrediction`
  - CatBoost ONNX risk model
  - operational fallback
  - people-aware forecast
- `src/services/regressionHistory.ts` используется из prediction layer для baseline.
- [[Google Apps Script и Drive]] используется через `GOOGLE_APPS_SCRIPT_URL` для исторических данных.
- [[Supabase]] через `src/services/dashboardSupabase.ts`:
  - `loadDashboardSnapshotHistory`
  - `saveDashboardSnapshot`
  - append-only история в `public.dashboard_snapshots`

## Настройки

- `allureBase`
- `allureToken`
- `projectId`
- `proxyBase`, `proxyMode`, `useProxy`
- `mlHelperBase` косвенно влияет на доступность ML-потока.

## Что показывает

- readiness по Android/iOS;
- группы Allure: Critical, Smoke, Selective, Regression и связанные с ними счетчики;
- остаток, in progress, assigned activity;
- предупреждения по пустым или подозрительным запускам;
- прогноз дедлайна, риск и причины риска;
- историю снимков релиза.

## Layout

Dashboard работает в wide-режиме shell:

- в `src/App.tsx` модуль `dashboard` входит в `isWideModule`;
- общий контейнер не ограничивает dashboard `maxWidth: 1440`;
- padding для dashboard уменьшен до `12px 14px 18px`;
- root dashboard занимает `width: 100%`;
- верхние metric cards используют `repeat(4, minmax(220px, 1fr))`;
- секция bar chart + alerts занимает всю ширину: основной график растягивается, alerts получает колонку `360..460px`;
- в карточке "Сводный барчарт" под графиком есть компактный `SummaryBarInsights`: узкие progress-строки по группам, самый большой хвост и самая нагруженная группа без раздувания высоты карточки;
- canvas сводного барчарта живет в фиксированном контейнере `280px`, а секция bar chart + alerts использует `alignItems: start`, чтобы карточка графика не растягивалась по высоте алертов;
- секция детализации + prediction также растянута на всю ширину, prediction держит минимальную читаемую ширину;
- "Детализация по 4 группам" растягивается по высоте соседнего блока прогноза и использует compact-режим карточек/донатов.

Цель: на широких экранах использовать всю доступную область после sidebar и показывать больше данных без пустых полей по краям.

## Полная логика работы

Основной вход - `Dashboard.load()` в `src/modules/Dashboard/index.tsx`.

1. Проверка настроек:
   - обязательны `allureBase`, `projectId`, `allureToken`;
   - из настроек собирается `cfg` для Allure: base, token, projectId, proxyBase, proxyMode, useProxy, abort signal.
2. Чтение истории:
   - сначала вызывается `loadDashboardSnapshotHistory(version, projectId, 48)` из Supabase;
   - параллельно вызывается `loadLatestDashboardSnapshot(version, projectId)`, чтобы восстановить полный UI без нажатия "Собрать dashboard";
   - если Supabase вернул историю, она становится основной `history`;
   - если Supabase недоступен или история пустая, используется localStorage `rp_dashboard_history_v1`;
   - `previousSnapshot` берется как последний срез из основной истории, иначе из `rp_dashboard_last_v1`.
3. Восстановление сохраненного состояния:
   - если `loadLatestDashboardSnapshot` вернул строку, UI получает `launches`, `agg`, `uwu`, `readiness`, `alerts`, `prediction` из payload этой строки;
   - это позволяет показать карточки, графики, readiness, alerts и prediction без ручного сбора;
   - если полного snapshot в Supabase нет, используется локальный dashboard cache;
   - если нет и локального cache, состояние графиков очищается под текущую версию.
4. Быстрый показ кеша:
   - `readCachedDashboardAggregate(version)` может сразу показать последний локальный агрегат;
   - `readCachedDashboardReadiness(version)` может сразу показать readiness;
   - если кеш есть, UI обновляется до завершения полного refresh;
   - prediction по кешу строится на уже загруженной истории Supabase/localStorage.
5. Полный сбор Allure:
   - параллельно запускаются `fetchDashboardAggregate(cfg, version, { forceRefreshDetails: true })` и `fetchReadinessSummary(cfg, version)`;
   - aggregate собирает релевантные release launches, exact statistics, memberstats, leaf data и uWu;
   - readiness отдельно находит готовностные launches Android/iOS;
   - readiness дополнительно кешируется через `writeCachedDashboardReadiness`.
6. Формирование runtime UI state:
   - `launches` получает список релизных launches;
   - `agg` получает counts по группам `DASHBOARD_ORDER`;
   - `uwu` получает High/Blocker uWu counters;
   - `alerts` получает пустые launches и launches без passed;
   - `readiness` получает Android/iOS готовность.
7. Формирование snapshot:
   - `buildHistorySnapshot()` превращает текущие aggregate/readiness/alerts/uWu в `DashboardHistoryPoint`;
   - snapshot сохраняется локально через `writeSnapshot()` как fallback;
   - локальная история, предыдущая Supabase-история и новый snapshot объединяются через `mergeHistoryPoints()`.
8. Прогноз:
   - `buildDashboardPrediction()` получает текущий aggregate, uWu, readiness, alerts, историю, deadline и данные активности людей;
   - движок использует CatBoost ONNX, если модель доступна, иначе operational fallback;
   - historical baseline подтягивается через `regressionHistory.ts` и Google Apps Script;
   - результат сохраняется в `prediction`.
9. Запись Supabase:
   - после прогноза вызывается `saveDashboardSnapshot()`;
   - запись append-only: каждый успешный сбор создает новую строку в `dashboard_snapshots`;
   - в строку пишутся плоские метрики, сырые payload, `history_point`, `prediction_payload`, `tracking_payload`, `enrichment_payload`;
   - после успешной записи UI перечитывает последние snapshots из Supabase и обновляет `history`.
10. Статус persistence:
   - рядом с автообновлением показывается `storageStatus`;
   - зеленый статус означает успешное чтение/запись Supabase;
   - желтый статус означает fallback или ошибку записи, при этом dashboard продолжает работать локально.
11. Автообновление:
   - чекбокс запускает интервал 10 минут;
   - по таймеру повторно вызывается текущий `loadRef.current()`;
   - каждый автосбор также пишет новый Supabase snapshot.

## Расчеты внутри snapshot

`DashboardHistoryPoint` хранит нормализованную точку истории:

- `total`, `finished`, `remaining` - суммарные кейсы по группам dashboard;
- `manualFinished`, `manualTimedFinished` - ручное прохождение, включая окно прохождения;
- `assigned`, `inProgress` - распределение остатка по memberstats;
- `readinessAndroid`, `readinessIos` - readiness pct по платформам;
- `criticalTotal`, `criticalFinished`, `selectiveTotal`, `selectiveFinished` - раздельные счетчики Critical/Selective;
- `emptyAlerts`, `noPassedAlerts` - качество launch данных;
- `uwuTotal`, `uwuLeft` - uWu остаток High/Blocker;
- `activePeopleCount`, `activePeopleLogins` - активные исполнители из Allure.

`tracking_payload` в Supabase дополнительно считает:

- дельту total/finished/remaining/manual к прошлому snapshot;
- elapsed hours между snapshot;
- manual velocity per hour;
- completion pct, manual completion pct;
- critical/selective completion pct;
- min readiness и readiness gap;
- compact alerts/uWu counters.

## Локальное состояние

- `rp_dashboard_last_v1`
- `rp_dashboard_history_v1`
- `rp_dashboard_form_v1` - последние введенные `version` и `customDeadline`
- кеши внутри `allure.ts`, включая release cache и readiness cache.

Локальное состояние теперь fallback. Основной источник истории dashboard - Supabase `dashboard_snapshots`.

## Supabase snapshots

Каждый успешный сбор dashboard пишет новый snapshot в `public.dashboard_snapshots`:

- плоские метрики для быстрых фильтров и сортировки;
- `history_point` для восстановления `DashboardHistoryPoint`;
- `aggregate_payload`, `uwu_payload`, `readiness_payload`, `launches_payload`, `alerts_payload`;
- `prediction_payload`;
- `tracking_payload` с дельтами относительно прошлого среза, completion/readiness pct и velocity;
- `enrichment_payload` с краткой сводкой источника и прогноза.

При смене версии и при старте сбора UI сначала читает последние 48 срезов из Supabase. Если Supabase недоступен, dashboard откатывается на localStorage.

## Таблица dashboard_snapshots

Таблица создается миграцией `supabase/dashboard_snapshots.sql`.

Назначение строки: один immutable snapshot dashboard для пары `project_id + version` в момент `snapshot_at`.

Основные группы колонок:

| Группа | Колонки | За что отвечает |
|---|---|---|
| Идентификация | `id`, `project_id`, `version`, `snapshot_at`, `source` | Уникальная точка истории dashboard. |
| Общие counts | `total_cases`, `finished_cases`, `remaining_cases`, `assigned_cases`, `in_progress_cases`, `launches_count` | Быстрый просмотр прогресса без разбора JSON. |
| Manual activity | `manual_finished_cases`, `manual_timed_finished_cases`, `active_people_count`, `active_people_logins` | Темп ручного прохождения и кто реально активен. |
| Readiness | `readiness_android`, `readiness_ios`, `readiness_min_pct`, `readiness_gap_pct` | Сравнение готовности платформ и разрыв между ними. |
| Critical/Selective | `critical_total`, `critical_finished`, `selective_total`, `selective_finished`, `critical_completion_pct`, `selective_completion_pct` | Раздельная аналитика критического пути и selective. |
| uWu | `uwu_total`, `uwu_left` | Остаток uWu по High/Blocker. |
| Alerts | `empty_alerts`, `no_passed_alerts` | Контроль качества launch данных. |
| Prediction | `prediction_status`, `prediction_risk`, `prediction_confidence`, `eta_at`, `deadline_at` | Быстрые поля прогноза без разбора `prediction_payload`. |
| Full payload | `history_point`, `aggregate_payload`, `uwu_payload`, `readiness_payload`, `launches_payload`, `alerts_payload`, `prediction_payload` | Полное восстановление состояния dashboard и отладка расчетов. |
| Enrichment/tracking | `tracking_payload`, `enrichment_payload` | Производные метрики, дельты, velocity, краткая сводка источника и прогноза. |
| Audit | `created_at`, `updated_at` | Технические даты строки. |

Индексы:

- unique `project_id + version + snapshot_at` защищает от дубля одного и того же snapshot;
- `version + snapshot_at desc` ускоряет просмотр истории версии;
- `project_id + version + snapshot_at desc` используется `loadDashboardSnapshotHistory`;
- GIN по `tracking_payload` оставлен для будущих аналитических фильтров.

RLS:

- `select` открыт для `anon` и `authenticated`;
- `insert` открыт для `anon` и `authenticated`;
- `update` открыт для `anon` и `authenticated`;
- delete policy нет, чтобы случайно не удалить историю из frontend.

## Что важно при рефакторинге

Не смешивать три разных расчета: сырые Allure counts, readiness summary и predictive risk. Безопасная граница - сначала вынести pure model для `DashboardHistoryPoint` и `DashboardPredictionInput`, а UI оставить прежним.

## Узкие места загрузки

Текущий сбор в `Dashboard.load()` показывает кэшированный срез сразу, но затем запускает полный refresh:

- `fetchDashboardAggregate(cfg, version, { forceRefreshDetails: true })`;
- `fetchReadinessSummary(cfg, version)`;
- внутри aggregate: `fetchLaunches`, затем по каждому релизному launch параллельно `statistic`, `memberstats`, `leaf`;
- concurrency detail-запросов задаёт `DASHBOARD_LAUNCH_DETAIL_CONCURRENCY`, сейчас `18`;
- для High/Blocker дополнительно считается uWu через `testcase/{id}/overview` по leaves;
- concurrency uWu overview задаёт `DASHBOARD_UWU_CASE_CONCURRENCY`, сейчас `20`.

`forceRefreshDetails: true` нужен для актуальной истории, но он отключает переиспользование detail cache и делает каждый refresh тяжелее.

## Варианты ускорения

1. Smart refresh вместо полного refresh: обновлять detail только для новых, изменившихся или RUNNING launch, а для DONE/stable брать cache. Периодически делать полный refresh вручную или по TTL.
2. Разделить быстрый и полный режим: быстрый dashboard строить по preview/statistics, а memberstats/leaves/uWu догружать вторым фоном и обновлять карточки постепенно.
3. Убрать дубли readiness: использовать уже найденные launches или общий launch-search cache, чтобы readiness не запускал отдельный поиск там, где это можно связать с dashboard aggregate.
4. Оптимизировать leaf/uWu: считать uWu только для High/Blocker и только при изменении previewKey; leaves для manual activity ограничивать активными/изменившимися launch.
5. Вынести сбор в backend/gateway: браузер делает один запрос к агрегатору, а сервер параллелит Allure, кеширует результаты и возвращает готовый DTO.
6. Добавить timing telemetry: логировать длительность `launches`, `readiness`, `stats`, `memberstats`, `leaves`, `uwu`, `prediction`, чтобы оптимизировать по фактическому bottleneck.
