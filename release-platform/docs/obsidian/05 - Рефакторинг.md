# Рефакторинг

## Текущее состояние

Release Platform уже разложен по UI-модулям, но это не независимые микрофронты в смысле Module Federation. Сейчас это единая React SPA, где микрофронт = экран из `src/modules`, подключенный shell через `src/App.tsx`.

Основная сложность не в UI, а в сервисном слое:

- HTTP-клиенты, агрегация и форматирование часто лежат в одном файле.
- Токены хранятся в браузере и синхронизируются со старыми legacy ключами.
- Часть persistence живет в `localStorage`, часть в Google Apps Script/Drive, часть в Supabase.
- Legacy HTML еще нужен как fallback и как источник отдельных assets.

## Приоритеты безопасного рефакторинга

1. Зафиксировать контракты данных для каждого микрофронта: input settings, output rows, error model.
2. Разделить service modules на `client`, `mapper`, `domain aggregate`.
3. Вынести работу с tokens/settings в единый adapter, чтобы модули не знали legacy keys.
4. Стабилизировать proxy contract: `query` и `prefix` режимы, cookie forwarding, health endpoints.
5. Перенести секреты и CORS-sensitive API в backend/gateway, когда появится серверная часть.
6. Для больших модулей начать с read-only refactor: [[Графики]], [[Запуск релиза]], [[Анализ релизов за квартал]].
7. Общую навигацию развивать через [[UI Navigation Kit]], не копируя sidebar-разметку по фронтам.

## Рекомендуемые границы

| Слой | Что должно жить внутри |
|---|---|
| `src/modules/*` | UI state, form state, вызов domain services, rendering |
| `src/services/*Client.ts` | Чистые HTTP-клиенты внешних API |
| `src/services/*Mapper.ts` | Нормализация payload внешних систем |
| `src/services/*Report.ts` | Доменная агрегация и расчеты |
| `src/context/*` | Runtime settings, theme, auth/session |
| `src/components/navigation/*` | Переиспользуемые navigation primitives и sidebar kit |
| `src/navigation/*` | Navigation data adapters, mock API и будущие clients |
| backend/gateway | Секреты, CORS, cookie exchange, server-side jobs |

## Navigation kit как первый adapter-layer

Текущий sidebar уже разделён на:

- `NavigationSidebar` - чистый UI-kit компонент.
- `Sidebar` - adapter текущего shell и RBAC-visible модулей.
- `sidebarMockApi` - временный mock API через fetch interceptor.

Это целевая форма для похожих зон: общий UI получает декларативный контракт, а конкретный фронт отвечает за свои ссылки, иконки, доступы и обработчик переходов.

## Первыми документировать перед изменениями

- [[Графики]]: самый широкий blast radius, много источников и ML/AI.
- [[Запуск релиза]]: побочные эффекты во внешних системах Band/Allure.
- [[ЧП по стримам]]: связывает Deploy Lab, YouTrack, GitLab и Drive.
- [[Wiki Intelligence v4]]: LLM, web search, wiki publish.
- [[UI Navigation Kit]]: общая навигация, mock API и правила подключения sidebar в новые фронты.

## Что не ломать

- `legacy/*` и имена HTML-файлов с кириллицей: Vite plugin и query `?module=` завязаны на эти имена.
- Существующие `localStorage` ключи: они используются для обратной совместимости с legacy.
- `proxyMode`: оба режима `query` и `prefix` поддержаны в `proxy.ts`.
- Navigation endpoints пока замоканы; при появлении реального API заменить adapter, а не UI-kit.
